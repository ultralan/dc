# 代码实现轮子与最佳实践建议

## 1. 结论先行

如果现在要把 `8` 麦克风均匀圆阵的音频侧融合模型真正做成代码，我建议采用下面这套技术栈：

### 1.1 必选轮子

- `Python 3.11`
- `uv`
- `torch`
- `torchcodec`
- `torchaudio`
- `numpy`
- `scipy`
- `lightning`（只用 `lightning.fabric`，不强依赖 Trainer）
- `hydra-core`
- `torchmetrics`
- `rich`

### 1.2 可选轮子

- `pyroomacoustics`：只用于仿真、阵列几何验证、SRP/DOA baseline，不放进训练热路径
- `wandb` 或 `mlflow`：二选一做实验追踪
- `tensorboard`：如果团队只想要本地最轻量追踪

### 1.3 开发质量轮子

- `pytest`
- `ruff`
- `ty` 或 `mypy`

### 1.4 不建议作为主链路核心依赖

- `librosa`
- 同时混用多套音频 IO 库
- 为了训练流程而上完整 `Lightning Trainer`

原因很简单：

- 这类模型是“音频 DSP + 空间几何 + 自定义多任务训练循环”的混合工程，核心要求不是“框架越多越好”，而是“训练主链路尽量短、张量尽量留在 PyTorch 里、配置和实验可复现”。

## 2. 为什么这样选

### 2.1 `torch` + `torchcodec` + `torchaudio` 是主干，不建议拆开

推荐原因：

- PyTorch 官方安装页当前把 `torch torchvision torchaudio` 作为标准安装组合提供。
- TorchAudio 官方安装文档明确强调：`torchaudio` 和 `torch` 必须使用匹配版本，不能混装不同 release。
- TorchAudio 官方文档当前已经说明：自 `2.8` 起进入 maintenance phase，音频解码/编码能力正在并入 `TorchCodec`，并推荐代码逐步迁移到原生 `torchcodec` API。
- 你的任务本质上需要：
  - 多通道张量处理
  - STFT/时频特征
  - GPU 训练
  - 自定义 loss
  - 流式推理

这些都更适合在同一套 PyTorch 张量生态里完成。

最佳实践：

- `torch` 和 `torchaudio` 固定在同一个 release line。
- 不要出现 `torch==A` 配 `torchaudio==B` 的情况。
- 安装时优先走 PyTorch 官方 wheel index，而不是自己混拼第三方源。
- 音频 I/O 优先 `torchcodec`，时频变换和音频 DSP 保持在 `torch/torchaudio` 体系内。

## 3. 环境与依赖管理

### 3.1 推荐 `uv`，不推荐手写一堆 `requirements.txt`

推荐原因：

- `uv` 官方支持项目环境、锁文件和自定义 package index。
- 对 PyTorch 这种需要官方 wheel index 的项目，`uv` 的 `[[tool.uv.index]]` 和 `tool.uv.sources` 很适合。

最佳实践：

- 用 `pyproject.toml` 作为单一依赖入口。
- 用 `uv lock` 固定依赖。
- 用 `uv sync` 保证开发机、CI、训练机一致。

建议环境基线：

- Python：`3.11`

这里的 `3.11` 是工程推荐值，不是官方唯一要求。我的判断依据是：

- PyTorch 安装页当前说明 latest stable 需要 `Python 3.10+`
- TorchAudio 安装页的兼容矩阵显示官方二进制分发支持到较新的 Python 版本

但从工程稳定性上，`3.11` 目前通常是更稳妥的折中点。

### 3.2 推荐的 `pyproject.toml` 依赖分层

```toml
[project]
name = "uca8-tracktrend"
version = "0.1.0"
requires-python = ">=3.11,<3.13"
dependencies = [
  "torch",
  "torchcodec",
  "torchaudio",
  "numpy",
  "scipy",
  "hydra-core",
  "lightning",
  "torchmetrics",
  "rich",
]

[project.optional-dependencies]
sim = ["pyroomacoustics"]
exp = ["wandb"]

[dependency-groups]
dev = [
  "pytest",
  "ruff",
  "ty",
]
```

如果团队更偏保守，可以把 `ty` 换成 `mypy`。

### 3.3 `uv` 下的 PyTorch index 写法

建议把 PyTorch 官方 index 单独声明，并显式绑定 `torch`/`torchaudio` 来源：

```toml
[tool.uv.sources]
torch = { index = "pytorch-cu126" }
torchaudio = { index = "pytorch-cu126" }

[[tool.uv.index]]
name = "pytorch-cu126"
url = "https://download.pytorch.org/whl/cu126"
explicit = true
```

如果是 CPU 机或 Windows 无 CUDA 需求，就换成对应的 CPU index。

关键点：

- `explicit = true` 很重要，避免别的包错误地从 PyTorch index 安装。
- `torch` 和 `torchaudio` 必须绑同一个 index。

## 4. 训练主链路推荐用法

### 4.1 频谱、重采样、音频加载：`torchcodec` 负责 I/O，`torchaudio` 负责变换

推荐：

- `torchcodec.decoders.AudioDecoder`
- `torchaudio.transforms.Resample`
- `torch.stft` / `torchaudio.transforms.Spectrogram`

不推荐把训练主链路放在：

- `librosa.load`
- `librosa.stft`
- `soundfile + numpy + librosa` 混搭

原因：

- TorchAudio 官方文档已经把 `load()` / `save()` 迁移到底层调用 `TorchCodec`，并建议直接迁移到 `torchcodec` 原生 API。
- `librosa` 更适合离线分析和快速原型，不适合作为训练热路径主库。
- 多通道 GPU 训练时，最好尽量少在 `numpy` 和 `torch` 之间来回搬数据。
- `torchaudio` 与 `torch` 同生态，后续混合精度、分布式、导出更顺。

我的建议是：

- 音频读取、分段解码、未来扩展到视频带音轨输入时，优先 `torchcodec`
- 训练/推理主链路中的 STFT、Mel、Resample、频域增强，使用 `torch + torchaudio`
- 只有在排查某些特殊音频格式兼容问题时，才临时引入 `soundfile`

## 5. 几何、匹配、后处理

### 5.1 `numpy` + `scipy` 是几何与传统算法层的标准组合

推荐原因：

- `scipy.optimize.linear_sum_assignment` 直接可用于多源槽位匹配 / Hungarian matching。
- `numpy` 适合做阵列坐标、方位角网格、离线标签生成。

推荐使用场景：

- 阵列几何定义
- DOA 网格生成
- 标签构造
- Hungarian 匹配
- 经典滤波和插值

最佳实践：

- 数据预处理和标签生成可以用 `numpy/scipy`
- 训练前向和 loss 计算尽量留在 `torch`

也就是说：

- `numpy/scipy` 负责“离线几何与标签”
- `torch` 负责“在线数值计算与梯度传播”

### 5.2 `pyroomacoustics` 适合离线，不适合热路径

推荐它的原因：

- 官方文档明确支持 room simulation、microphone array、source placement。
- 官方文档提供了 `SRP` DOA 类。
- 官方文档里也有现成的 `circular_2D_array` 等规则阵列生成能力。

但我不建议把它放到训练主链路里，原因是：

- 它更适合“数据合成、baseline 验证、算法 sanity check”
- 训练和推理主链路里，自己用 `torch` 实现 SRP/IPD 特征会更好控、更快、更容易上 GPU

最佳实践：

- 用 `pyroomacoustics` 做：
  - 8 麦圆阵几何验证
  - 无混响/有混响仿真数据生成
  - SRP-PHAT baseline
- 不要让训练 dataloader 每个 batch 都临时调用 `pyroomacoustics`

## 6. 训练框架选型

### 6.1 推荐 `lightning.fabric`，不推荐直接上 `Lightning Trainer`

这是这类项目里最关键的一个选择。

推荐原因：

- Lightning 官方文档明确把 `Fabric` 定位为“轻量、少 boilerplate、保留训练循环控制权”的方案。
- 文档也明确说：Fabric 适合不想把代码重构成标准化 Trainer 形态、但又想要 DDP/FSDP/mixed precision/checkpoint/logging 的场景。

这恰好符合你的模型特点：

- 多任务损失
- 当前状态头 + 未来趋势头
- 流式窗口训练
- 可能需要自定义 evaluation loop
- 可能需要在线 Kalman 后处理

如果直接用 `Lightning Trainer`，通常会在下面这些点变得更别扭：

- 自定义时间展开逻辑
- 复杂验证指标
- 流式推理逻辑和训练逻辑共存
- 逐步插入传统后处理模块

最佳实践：

- 模型、loss、metrics 仍然用纯 `torch`
- 训练脚本用 `lightning.fabric` 接管：
  - device
  - amp
  - distributed
  - checkpoint
  - logger

换句话说：

- 训练范式：`raw PyTorch + Fabric`
- 不建议：`raw PyTorch + 手写所有分布式细节`
- 也不建议：`一开始就 Trainer 化`

## 7. 配置管理

### 7.1 推荐 `hydra-core`

推荐原因：

- Hydra 官方文档对 Config Group、Defaults List、命令行 override 这些能力已经非常成熟。
- Structured Config 支持 dataclass，并支持运行时和静态类型检查。
- 很适合这类有大量实验维度的项目：
  - 阵列几何
  - 特征参数
  - 模型宽度
  - loss 权重
  - 数据划分
  - 训练策略

最佳实践：

- 用 dataclass 做 Structured Config
- `model/`, `data/`, `train/`, `feature/`, `runtime/` 分目录管理
- 训练脚本只接收一个总 config

建议配置层级：

```text
configs/
  config.yaml
  model/
    uca8_tracktrend.yaml
  data/
    locata_like.yaml
  feature/
    stft_16k.yaml
  train/
    base.yaml
  runtime/
    gpu_single.yaml
    gpu_ddp.yaml
```

不建议：

- 在脚本里硬编码超参数
- 用多个不相关的 yaml 文件手动拼
- CLI 参数和 yaml 参数各管一半

## 8. 指标与日志

### 8.1 推荐 `torchmetrics`

推荐原因：

- 官方文档强调它是分布式兼容、可自动跨 batch 累积的统一 metrics API。
- 对多卡训练比手写均值逻辑更稳。

建议用途：

- 源数量准确率
- F1
- MAE / RMSE
- 自定义 angular MAE metric
- 未来趋势准确率

最佳实践：

- 通用分类/回归指标用 `torchmetrics`
- 声源方位角误差、轨迹断裂率这类任务特有指标自己封装成 custom metric

### 8.2 实验追踪推荐二选一

如果团队需要多人协作和远程看板：

- `wandb`

如果团队更偏本地或内网自建：

- `mlflow`

如果现在只是单机开发：

- 先 `tensorboard`

我的建议：

- 个人原型：`tensorboard`
- 团队迭代：`wandb` 或 `mlflow` 二选一，不要同时上两套

## 9. 工程质量工具

### 9.1 测试：`pytest`

推荐原因：

- 官方文档强调它的 assertion 简洁、fixture 机制成熟、适合从小测试到复杂测试。

至少要写的测试：

- 阵列几何测试
- 标签生成测试
- SRP/IPD 形状测试
- loss 输入输出测试
- 流式推理窗口拼接测试

### 9.2 lint / format：`ruff`

推荐原因：

- Ruff 官方文档明确说它可以替代 Flake8、Black、isort 等多类工具，且速度极快。

最佳实践：

- 统一用 Ruff 做 lint + format
- 不要再并行维护 `black + isort + flake8`

### 9.3 类型检查：`ty` 或 `mypy`

如果你追求速度和新工具体验：

- `ty`

如果你追求生态稳定：

- `mypy`

我的建议：

- CI 首选 `mypy`
- 本地开发可尝试 `ty`

原因不是功能绝对差异，而是工程成熟度权衡。

## 10. 不建议的做法

### 10.1 不建议把 `librosa` 放进训练主路径

原因：

- 多通道 + GPU + streaming 场景下，`librosa` 不是最顺手的主干库。
- 容易造成 `numpy`/`torch` 来回搬运。

### 10.2 不建议为了“看起来高级”堆太多框架

不建议一上来同时用：

- Lightning Trainer
- Hydra
- MLflow
- W&B
- Ray Tune
- DVC

这会让 MVP 阶段的复杂度爆炸。

更合理的顺序是：

1. `uv + torch + torchcodec + torchaudio + scipy`
2. 再加 `hydra`
3. 再加 `fabric`
4. 最后加 `wandb/mlflow`

### 10.3 不建议让传统 DSP 库进入热路径

例如：

- 训练时每个 batch 动态调用 `pyroomacoustics`
- 在 dataloader 中大量做 Python 层循环几何计算

最佳实践是：

- 离线预计算的尽量预计算
- 在线必须算的尽量 `torch` 化

## 12. 最终建议

如果你的目标是“尽快做出可训练、可验证、可迭代的 8 麦圆阵音频模型”，最佳实践不是选最多的轮子，而是选最少且边界清晰的一组：

- 环境与锁版本：`uv`
- 深度学习主干：`torch + torchcodec + torchaudio`
- 几何与匹配：`numpy + scipy`
- 训练工程化：`lightning.fabric`
- 配置：`hydra-core`
- 指标：`torchmetrics`
- 质量保障：`pytest + ruff`
- 仿真与 baseline：`pyroomacoustics`

这是我认为最稳、最容易维护、也最符合你当前任务形态的实现组合。
