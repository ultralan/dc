# 论文完成执行计划

> **执行原则**：先把论文结论需要的证据链跑出来，再写正文。每个阶段都必须产出可复用文件、指标表或图，不以“感觉差不多”作为完成标准。

**目标**：完成一篇以“通用麦克风阵列模型 + 8 麦圆阵树莓派真实设备实验”为主线的声源定位、跟踪与短窗趋势预测论文。

**主线**：提出一个几何可配置的多任务时序模型，用可解释的阵列空间表示连接真实多通道音频与当前定位、同类多源槽位、未来短窗趋势预测；在 RealMAN 子阵列和 8 麦树莓派平台上验证准确性、泛化性和边缘部署潜力。

**现有基础**：仓库已有模型骨架、RealMAN 数据读取、训练脚本、基础验证指标、文献调研和论文故事线。缺口主要在数据分布画像、baseline 对比、阵列泛化实验、树莓派部署实测、论文正文与图表。

## 5. 阶段 4：阵列泛化与通用模型证据

**目标**：把“通用模型”从口号变成实验证据，同时避免超出当前代码能力。

**产物**：

- 新增配置：`configs/model/geometry_configurable.yaml`
- 新增配置：`configs/train/realman_array_generalization.yaml`
- 新增结果表：`outputs/paper/tables/table_array_generalization.md`
- 如果实现结构改造，新增测试：`tests/test_variable_array_model.py`

**两档方案**：

### 方案 A：保守版，当前代码可优先完成

当前 `IPDEncoder(in_channels=24)` 偏 8ch 固定。保守版不宣称“任意通道数直接输入”，只做：

- fixed `ring2_8ch` 训练与测试。
- `mic dropout` 训练，在同一 8ch 几何下测试鲁棒性。
- 不同 8ch 子阵列或 unseen split 测试。

论文表述：

```text
The framework is geometry-configurable, and this paper empirically studies its adaptation to an 8-channel real-device setting.
```

### 方案 B：增强版，真正支撑 variable-channel

把 IPD 编码改成 pair-wise shared encoder：

```text
每个麦对共享同一个小 CNN/MLP
  -> 对 pair embedding 做 mean/max/attention pooling
  -> 得到固定维度 IPD 表示
```

这样可支持不同麦克风数量和不同麦对数量。

**建议先走方案 A**，等主实验稳定后再决定是否投入方案 B。

**验收标准**：

- 至少有 fixed 8ch vs mic dropout 的对比。
- 如果没有完成方案 B，正文中不用“任意通道数通用”这种强表述。
- 所有泛化实验都有配置文件和结果 JSON。

## 6. 阶段 5：树莓派边缘部署实验

**目标**：实测模型在树莓派上的运行可行性，明确是离线可跑、准实时可跑，还是需要量化和流式优化。

**产物**：

- 新建 `scripts/benchmark_edge_inference.py`
- 新建 `outputs/paper/edge/raspberry_pi_benchmark.json`
- 新建 `outputs/paper/tables/table_edge_deployment.md`
- 如果导出模型，新增 `exports/uca_tracktrend_fp32.onnx`
- 如果量化，新增 `exports/uca_tracktrend_int8.onnx`

**实验平台**：

| 平台               | 精度      | 目的             |
| ------------------ | --------- | ---------------- |
| Mac M1 Pro CPU     | FP32      | 开发机参考       |
| Raspberry Pi 5 CPU | FP32      | 原始模型可运行性 |
| Raspberry Pi 5 CPU | INT8/ONNX | 实际部署潜力     |
| Raspberry Pi 4 CPU | INT8/ONNX | 可选低配验证     |

**指标**：

| 指标     | 解释                                    |
| -------- | --------------------------------------- |
| 参数量   | 当前粗估约 `7.56M`                    |
| 模型大小 | FP32 约 `28.9 MB`，INT8 约 `7.2 MB` |
| 单窗延迟 | 一次 1.28s 历史窗口推理耗时             |
| RTF      | 推理时间 / 音频窗口长度                 |
| 峰值内存 | 推理期间最大内存                        |
| 前端占比 | STFT/IPD/SRP 占总耗时比例               |

**建议命令**：

```bash
uv run python scripts/benchmark_edge_inference.py \
  --config configs/realman_ring2_mvp.yaml \
  --checkpoint runs/realman_ring2_mvp/20260610_120000/best.pt \
  --device cpu \
  --warmup 10 \
  --repeat 100 \
  --output outputs/paper/edge/mac_cpu_benchmark.json
```

树莓派端命令保持同样接口：

```bash
python scripts/benchmark_edge_inference.py \
  --config configs/realman_ring2_mvp.yaml \
  --checkpoint best.pt \
  --device cpu \
  --warmup 10 \
  --repeat 100 \
  --output raspberry_pi_benchmark.json
```

**验收标准**：

- 能给出明确表述：离线可跑、准实时可跑，或需要量化/流式优化。
- 不在论文中空口写“实时运行”。
- 如果 RTF 小于 `1.0`，可以写具备实时潜力；如果 RTF 大于 `1.0`，写当前模型可部署但实时性需要优化。
