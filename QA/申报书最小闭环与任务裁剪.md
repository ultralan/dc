# 申报书最小闭环与任务裁剪

## 当前结论

当前阶段不再把模型推进成“大而全”的多任务系统。申报书最需要兑现的是：

```text
噪声智能预测 + 声源定位/溯源 + 声像协同展示 + LoRa/边缘系统链路
```

因此实验主线收缩为两层：

1. `baseline 对齐层`：只做当前声源方位定位，输出 MAE 和 ACC(5deg)。
2. `申报书展示层`：保留噪声预测、定位可视化、声像协同和 LoRa 系统说明。

## 先砍掉什么

这些暂时不作为主实验：

- 多源 slot 建模
- future slot 预测
- future heatmap 预测
- 任意阵列泛化
- 树莓派 INT8 实测
- 严格超过 RealMAN IPDnet

原因是这些会显著拖慢训练和实验闭环，而且不是申报书最核心的验收点。

## 必须保留什么

### 1. 当前方位定位

用于对齐 RealMAN baseline：

- `MAE [deg]`
- `ACC(5deg)`
- `static / moving` 分组

当前代码对应 `heatmap_logits`，评估以 `heatmap_argmax` 作为主要定位结果。

### 2. 噪声预测

用于兑现 `TCN 驱动的噪声智能预测`：

- 输入：历史噪声 dB 序列或声学特征序列
- 输出：未来噪声 dB
- 指标：MAE / RMSE / R2
- 图：预测曲线 vs 真实曲线

### 3. 声像协同溯源

最低可交付形式：

- 声源方位角
- 场景平面图或摄像头画面
- 方位箭头 / 热力图叠加

这比多源 slot 更贴申报书。

### 4. LoRa 系统链路

LoRa 不一定先做复杂通信实验，但必须有：

- 系统架构图
- 数据上传字段
- 上传频率/延迟/丢包说明
- 一组模拟或实测日志

## 当前代码策略

新增 `realman_ring2_localization_only` 配置：

- 保留现有模型结构，避免大拆引入 bug
- 关闭 count / slot / future / trend 损失权重
- 只集中优化当前 heatmap 定位头
- 训练后用 `evaluate_realman_baseline_metrics.py` 输出 RealMAN 风格指标

这个策略不是最终最优结构，但能最快得到可和 baseline 放在同一张表里的结果。

