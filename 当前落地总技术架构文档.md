# 当前落地总技术架构文档

## 1. 文档目的

本文档不是再写一版抽象方案，而是把当前项目中已经落地的代码、训练链路和论文依据统一起来，回答三个问题：

1. 当前系统到底在做什么
2. 当前实现为什么这样设计
3. 下一步应该沿着哪条技术主线继续做强

本文档以当前仓库实现为准，结合已整理的论文综述：

- `文献调研_声源定位跟踪/00_综述与方案摘要.md`
- `文献调研_声源定位跟踪/01-10_*.md`
- `文献调研_声源定位跟踪/11-14_*.md`

## 2. 任务定义与落地边界

### 2.1 当前任务

当前系统面向 `8` 麦克风圆形均匀阵列设备，围绕语音源做统一建模，输出分为两层：

- 当前状态
- 短窗未来状态

当前状态包括：

- 当前活动源数量 `count`
- 当前方位热力图 `heatmap`
- 固定槽位的源状态 `slot_state`

短窗未来状态包括：

- 未来 `32` 帧的源数量
- 未来 `32` 帧的方位热力图
- 未来 `32` 帧的槽位轨迹状态
- 主导运动趋势类别 `motion/trend`

### 2.2 当前不做的内容

当前系统明确不覆盖：

- 视频检测
- 振动信号融合
- 三维高精度距离定位
- 说话人身份识别
- 全局长时轨迹关联

当前版本聚焦于：  
`8` 麦音频输入下的 `source count + azimuth state + short-horizon future trend`。

## 3. 论文驱动下的设计原则

当前落地架构不是拍脑袋拼出来的，而是沿着 5 条文献线收敛得到的。

### 3.1 空间前端必须保留可解释先验

依据：

- Knapp & Carter 1976
- Valin et al. 2007

结论：

- 小阵列、多源、混响条件下，`GCC/TDOA/SRP-PHAT` 这类空间特征依然有价值
- 深度模型不应完全替代可解释空间先验

所以当前实现保留：

- `IPD`
- `SRP-PHAT`
- 显式阵列几何

### 3.2 VAD/activity 必须进入主模型

依据：

- Vecchiotti et al. 2019
- Pertilä et al. 2021

结论：

- 活动检测错误会直接污染定位和跟踪
- VAD 不能只作为离线预处理

所以当前实现将 `vad_history` 明确作为模型输入，并在标签侧生成 `vad_ratio`。

### 3.3 表示应从通用 SELD 转向 speech-specific 多实例建模

依据：

- Adavanne et al. 2019 SELDnet
- Shimada et al. 2021 ACCDOA
- Shimada et al. 2022 Multi-ACCDOA

结论：

- 对当前单类 speech 多实例问题，不需要大类别空间
- 重点应放在 `count / activity / azimuth / slot assignment / overlap`

所以当前实现采用：

- 共享编码器
- 多头输出
- 固定槽位状态表示

### 3.4 时序主干当前阶段选 TCN 是合理的

依据：

- Lea et al. 2017
- Bai et al. 2018
- WaveNet 2016
- Conv-TasNet 2019

结论：

- `TCN` 对中短时程音频序列建模是成熟路线
- 因果空洞卷积适合低延迟部署
- 在语音任务中，`TCN` 兼顾效果、训练稳定性和工程友好性

所以当前实现采用 `Causal TCN` 作为统一时序 backbone。

### 3.5 数据训练必须服务真实设备

依据：

- RealMAN 2024

结论：

- 真实录制数据优于纯仿真训练
- 子阵列训练可以服务未见阵列泛化

所以当前系统的数据主线是：

- 基于 `RealMAN`
- 固定抽取 `ring2` 的 `8ch`
- 后续再扩展为多子阵列训练

## 4. 当前总体技术架构

当前系统可以概括为：

```text
RealMAN ring2 8ch waveform
  -> 多通道 STFT
  -> 四类输入特征
     - logmel_ref
     - logmel_rms
     - IPD
     - SRP-PHAT
     - vad_history
  -> 三分支编码
     - 频谱分支
     - IPD 空间分支
     - SRP 空间分支
  -> 特征融合
  -> Causal TCN 时序主干
  -> 多任务头
     - 当前 count
     - 当前 heatmap
     - 当前 slot_state
     - future count
     - future heatmap
     - future slot_state
     - motion trend
  -> 训练损失 / 可视化 / 后处理
```

这条链路的核心思想是：

- 前端负责“把空间信息和活动信息显式暴露出来”
- 主干负责“统一建模短中时程状态演化”
- 输出头负责“拆解成当前状态和未来状态”

## 5. 数据层架构

### 5.1 数据来源

当前主数据集是 `RealMAN ring2 8ch` 子集，配置位于：

- `configs/data/realman_ring2.yaml`

当前固定通道为：

- `channel_ids: [9, 10, 11, 12, 13, 14, 15, 16]`

这对应 RealMAN 原始 `32ch` 阵列中的一组 `8ch` 子阵列。

### 5.2 数据样本构造

当前数据集类：

- `src/uca8/data/realman_ring2_dataset.py`

每条样本的构造流程为：

1. 读取 `8ch` 阵列音频和 direct-path 音频
2. 将位置 CSV 中的角度、距离、高度序列插值到帧级长度
3. 从 direct-path 音频计算简化 `VAD`
4. 将整段序列裁成：
   - 历史 `128` 帧
   - 未来 `32` 帧
5. 交给标签构造器生成监督目标

### 5.3 当前标签定义

标签构造器位于：

- `src/uca8/data/label_builder.py`

当前监督目标包括：

- `count`
- `vad_ratio`
- `heatmap`
- `slot_state`
- `future_count`
- `future_heatmap`
- `future_slot_state`
- `trend_class`

其中 `slot_state` 的每个槽位是 `5` 维：

- `activity`
- `sin(theta)`
- `cos(theta)`
- `rho`
- `omega`

这相当于把 ACCDOA 的活动耦合思想扩展到了：

- 角度
- 距离
- 角速度

### 5.4 当前标签分配策略

当前槽位分配逻辑是：

- 若环境中有 `scipy`，优先使用 Hungarian assignment
- 否则按角度排序退化分配

这说明当前系统已经意识到 `same-class overlap` 的 assignment 问题，但实现还处于第一版。

## 6. 阵列几何与空间建模

### 6.1 阵列几何

阵列几何定义位于：

- `src/uca8/geometry/uca8.py`

当前采用二维均匀圆阵：

- 麦克风数：`8`
- 默认半径：`0.045 m`
- `mic0` 位于 `x` 轴正方向

这与当前设备假设是一致的。

### 6.2 麦对选择

当前默认麦对包括两组：

- `8` 组相邻麦对
- `4` 组对径麦对

总计 `12` 对，最终产生：

- `24` 通道 `IPD` 特征

这种设计兼顾：

- 局部相位差稳定性
- 大基线方向分辨率

### 6.3 方位网格

当前热力图采用：

- `72` 个 azimuth bins
- 即每格 `5°`

这与当前 `SRP` 和 `heatmap` 输出一致。

## 7. 特征前端架构

特征前端位于：

- `src/uca8/features/stft.py`
- `src/uca8/features/ipd.py`
- `src/uca8/features/srp_phat.py`

### 7.1 STFT 配置

当前特征配置：

- 采样率：`16 kHz`
- `n_fft = 512`
- `win_length = 400`
- `hop_length = 160`

即：

- 窗长 `25 ms`
- 帧移 `10 ms`

### 7.2 四类主输入特征

当前前端输出 5 个张量：

- `logmel_ref`
- `logmel_rms`
- `ipd_feat`
- `srp_map`
- `vad_ratio`

它们的职责分别是：

- `logmel_ref`：参考通道谱信息
- `logmel_rms`：多通道整体能量统计
- `ipd_feat`：细粒度相位差信息
- `srp_map`：显式空间扫描热力图
- `vad_ratio`：活动先验

### 7.3 当前前端实现状态

需要明确的是，当前 `logmel_like` 仍是第一版近似实现：

- 先取功率谱
- 再做频率池化压缩

它还不是严格 mel filterbank。  
这意味着当前前端是“可训练、可跑通”的工程版，而不是最终版。

## 8. 模型架构

模型入口位于：

- `src/uca8/models/tracktrend_net.py`

### 8.1 整体结构

模型名：

- `UCA8TrackTrendNet`

主结构为：

```text
FeatureFrontend
  -> SpectrogramEncoder
  -> IPDEncoder
  -> SRPEncoder
  -> Fusion Linear + LayerNorm + GELU
  -> CausalTCN
  -> Current Heads + Future Heads + Motion Head
```

### 8.2 三个编码分支

#### 8.2.1 频谱分支

位于：

- `src/uca8/models/spec_encoder.py`

结构特点：

- `2D Conv + Residual2d`
- 逐步压缩频率维
- 输出 `h_spec in R^(B,T,128)`

职责：

- 建模语音谱结构
- 吸收参考通道与整体 RMS 的谱线索

#### 8.2.2 IPD 分支

位于：

- `src/uca8/models/spatial_encoder.py`

结构特点：

- `Conv2d`
- depthwise conv
- residual blocks
- 输出 `h_ipd in R^(B,T,64)`

职责：

- 建模细粒度的相位差空间关系

#### 8.2.3 SRP 分支

位于：

- `src/uca8/models/spatial_encoder.py`

结构特点：

- 两层角度卷积
- 线性投影
- 输出 `h_srp in R^(B,T,64)`

职责：

- 提供显式的角度扫描先验
- 辅助主干做多源位置判断

### 8.3 融合层

三分支输出与 `vad_ratio` 拼接后，经过：

- `Linear`
- `LayerNorm`
- `GELU`

映射到统一 `model_dim = 256`。

这一步对应论文上的“共享编码器表示”，也是多任务头的统一输入。

### 8.4 时序 backbone：Causal TCN

位于：

- `src/uca8/models/tcn_backbone.py`

当前配置：

- `model_dim = 256`
- `kernel_size = 3`
- `dilations = [1, 2, 4, 8, 16, 32]`
- 每个 dilation 重复两次
- `GLU-like gate`
- 残差连接
- `LayerNorm + Dropout`

#### 8.4.1 为什么当前选 TCN

基于 Bai 2018、WaveNet 2016、Conv-TasNet 2019，当前阶段选 `TCN` 的理由是：

- 适合 `128` 帧历史到 `32` 帧未来的短中时程任务
- 适合因果、低延迟场景
- 比 RNN 更易并行
- 比 Transformer 更省算力、更稳
- 对帧级多任务输出更自然

#### 8.4.2 TCN 的角色边界

当前必须明确：

- `TCN` 负责时序表征
- 不负责彻底解决多源 identity 问题

多源身份连续性仍需依赖：

- label assignment
- slot consistency
- 后处理 tracker

### 8.5 输出头

当前输出头位于：

- `src/uca8/models/heads.py`

输出包括 7 组：

- `count_logits`
- `heatmap_logits`
- `slot_logits`
- `future_count_logits`
- `future_heatmap_logits`
- `future_slot_logits`
- `motion_logits`

其中：

- 当前头使用主干最后一帧状态
- 未来头先经过 `FutureDecoder`
- `FutureDecoder` 当前是轻量 `Conv1d` 结构

这说明当前系统已经不是“单帧定位模型”，而是一个真正的“当前状态 + 未来状态”联合模型。

## 9. 损失函数与训练目标

损失位于：

- `src/uca8/losses/multi_task_loss.py`

当前总损失由 5 部分组成：

- `count_loss`
- `heat_loss`
- `track_loss`
- `future_loss`
- `motion_loss`

默认权重为：

- `count_weight = 1.0`
- `heat_weight = 1.0`
- `track_weight = 2.0`
- `future_weight = 1.5`
- `motion_weight = 0.5`

### 9.1 当前损失设计的意义

这组损失实际上对应了 3 类目标：

1. 当前状态估计
2. 未来状态预测
3. 动态趋势分类

这与论文综述中的最终方案是一致的，即：

- 不只做“当前定位”
- 还做“短窗未来演化”

### 9.2 当前损失的局限

当前 `slot_loss` 已经区分：

- activity
- regression

但还没有完整引入：

- Hungarian-aware loss
- PIT/ADPIT 式 future assignment
- angle continuity regularization

这部分是下一阶段最值得补的地方。

## 10. 训练控制面与工程化机制

训练脚本位于：

- `scripts/train.py`

配置体系位于：

- `configs/config.yaml`
- `configs/model/uca8_tracktrend.yaml`
- `configs/data/realman_ring2.yaml`
- `configs/feature/stft_16k.yaml`
- `configs/train/base.yaml`
- `configs/runtime/local.yaml`

### 10.1 当前训练框架

当前采用：

- `Hydra` 管理配置
- `lightning.fabric` 管理设备与训练包装
- `AdamW` 优化器
- 梯度裁剪
- checkpoint 保存
- 可恢复训练

### 10.2 已落地的工程机制

当前已经具备：

- 统一配置解析与落盘
- manifest cache
- audio cache
- train history 持续记录
- `last.pt / best.pt / step_xxx.pt`
- RNG 状态保存与恢复
- 防止旧 checkpoint 覆盖新 run 的保护逻辑

这些机制说明当前项目已经不只是一个实验 notebook，而是一条成形的训练流水线。

### 10.3 当前默认训练策略

当前基础配置更偏 demo/smoke test：

- `batch_size = 2`
- `max_epochs = 1`
- `limit_train_steps = 5`
- 默认 `run_validation = false`

因此当前系统已经跑通，但默认配置仍是“最小可运行配置”，不是最终训练配置。

## 12. 当前落地能力判断

### 12.1 已经落地的部分

当前已经真实落地并可运行的能力包括：

- 基于 RealMAN `8ch` 子阵列的数据读入
- 帧级标签自动构造
- `logmel + IPD + SRP + VAD` 前端
- 共享编码器 + Causal TCN 主干
- 当前状态与未来状态联合输出
- 多任务损失训练
- checkpoint/恢复/可视化工程链路

### 12.2 当前版本的真实定位

当前版本应被定义为：

`面向 8 麦圆阵语音场景的第一版可训练 TrackTrend MVP`

它已经能承担：

- 架构验证
- 数据链路验证
- 训练机制验证
- 可视化验证

但还不是最终设备版。

## 13. 当前主要技术风险

### 13.1 前端谱特征仍是近似版

当前 `logmel_like` 还不是严格 mel filterbank。  
这会限制谱特征表达质量。

### 13.2 多源 assignment 机制还不够强

虽然当前已经有槽位和 Hungarian 入口，但还没有真正完成：

- current/future 一致 assignment
- ADPIT/PIT 风格训练
- identity memory

### 13.3 未来预测头仍偏轻量

当前 `FutureDecoder` 是简单的卷积解码器。  
对于更复杂的源交叉和未来轨迹分叉，可能表达力不够。

### 13.4 数据仍以固定子阵列为主

当前主训练路径还是固定 `ring2_8ch`。  
如果要提升设备泛化，必须进一步做：

- 多子阵列训练
- 麦克风 dropout
- 几何扰动增强

### 13.5 当前验证配置仍偏 demo

默认配置不跑验证，这意味着：

- 当前工程更像开发态
- 还没完全切换到长期训练态

## 14. 下一阶段推荐演进路线

### 14.1 第一优先级：补强训练正确性

建议优先做：

- 真正 mel filterbank
- 默认打开 validation
- 更完整指标
- 更强 slot assignment

### 14.2 第二优先级：补强多源稳定性

建议继续做：

- Hungarian-aware slot loss
- future slot PIT
- 轨迹连续性正则
- Kalman 与主推理打通

### 14.3 第三优先级：补强设备泛化

建议引入：

- 多 `8ch` 子阵列采样
- 通道缺失增强
- 阵列半径/几何扰动
- 设备校准参数注入

### 14.4 第四优先级：补强未来预测头

在不推翻整体结构的前提下，可升级：

- `FutureDecoder`
- motion/trend 监督
- future occupancy/VAD 监督

## 15. 总结

当前系统的技术路线已经比较清晰：

- 用 RealMAN 真实数据解决“能不能贴设备训练”的问题
- 用 `IPD + SRP + VAD` 解决“空间先验与活动先验”的问题
- 用共享编码器 + 多头输出解决“当前状态和未来状态统一建模”的问题
- 用 `Causal TCN` 解决“中短时程时序建模和部署友好性”的问题
- 用轻量 tracker 解决“输出平滑与工程稳定性”的问题

因此，当前最准确的架构定义是：

`一个以 RealMAN 8ch 子阵列为数据底座、以空间显式特征为前端、以 Causal TCN 为统一时序主干、以多任务输出实现当前定位与短窗未来趋势预测的 8 麦圆阵语音 TrackTrend 系统。`

这条路线和当前论文脉络是一致的，也与当前代码实现相匹配。  
后续优化重点应是“补强 assignment、验证和泛化”，而不是推翻整个架构重来。
